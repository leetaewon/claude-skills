# CHANGELOG — repozer

## 2026-06-17-003 (단계 분리 — 사용자 확인 턴 삽입)

### 변경 배경

2026-06-17-002에서 저장소 소속 필터링과 "동일 버전 존재 여부 확인" 절차를
도입했지만, 같은 대화에서 다시 오판이 발생했다. `leetaewon-brand-CLAUDE.md`
검색 결과에 본문 저장소명이 일치하지 않는 다른 저장소의 복제본
(`claude-context-design-book-CLAUDE.md`)이 섞여 있었는데, 제목이 비슷하다는
이유로 본문 확인 없이 "복제본이 있다"고 판단해 잘못 "동기화됨"으로
보고했다. `leetaewon-brand-index.md`도 같은 방식으로 원본 `index.md`를
복제본으로 착각했다.

근본 원인은 절차 자체가 아니라 실행 방식에 있었다 — 검색·필터링·비교·
실행이 한 턴 안에서 한 번에 처리되다 보니, 중간에 판단이 틀려도 사용자가
끼어들어 바로잡을 지점이 없었다. 사용자가 직접 4단계로 나누는 방안을
제안했다: ① 저장소 버전 확인 → ② 복제본 확인 → ③ 비교·제안 → ④ 사용자
승인 후 처리.

### 변경 내용

**Step 1을 세 개의 사용자 확인 턴으로 분리**
"진단 — 버전 비교"였던 Step 1을 "확인 — 버전 보고 및 비교 제안"으로
바꾸고, 1-1(저장소 버전) → 1-2(복제본 확인) → 1-3(비교·제안) 사이에
각각 사용자 응답을 기다리는 지점을 두었다. 이전에는 1-1과 1-2를 같은
턴에서 처리하고 1-3에서 결과만 한꺼번에 보여줬다.

**1-2 출력 형식에 본문 근거를 의무화**
"본문에 `[저장소명]-CLAUDE.md` 파일명이 명시된 청크: 있음/없음"과
"버전 태그: 있음/없음"을 출력 형식 자체에 못박았다. 제목만 보고
판단을 내릴 수 없도록, 판단의 근거가 되는 사실을 먼저 텍스트로
드러내게 했다. 제목만 비슷한 다른 청크가 섞여 들어온 경우 그 사실도
참고 항목으로 노출하도록 했다.

**1-3을 "판단"이 아닌 "제안"으로 전환**
1-1과 1-2에서 이미 보여준 사실을 토대로 상태(동기화됨/교체 필요/복제본만
없음/신규 생성 필요)를 "제안"하고, 사용자 동의를 받은 뒤에만 Step 2로
진행하도록 바꿨다. 이전에는 진단 결과를 보여주자마자 같은 턴에서 바로
Step 2 처리까지 이어졌다.

### 영향

- 검색 결과 오판이 실행 직전 사용자 확인 단계에서 발견될 수 있다.
- 본문 근거가 출력에 강제로 포함되어, 제목 유사성만으로 복제본 존재를
  단정하는 오류가 줄어든다.
- 단계가 늘어난 만큼 사용자 응답 횟수도 늘어난다 — 빠른 단일 턴 처리를
  원하는 경우에는 트레이드오프가 있다.

## 2026-06-17-002

### 변경 배경

`/repozer leetaewon-brand` 실행 중 두 가지 오판이 발생했다.

1. CLAUDE.md 복제본 검색 결과에 여러 버전(태그 없음 / 2026-06-09 /
   2026-06-16-002 / 다른 파일명의 옛 버전)이 동시에 섞여 나왔는데, 그중
   하나를 임의로 "복제본 버전"으로 골라 저장소 버전과 비교했다. 실제로는
   최신 버전(2026-06-16-002)과 동일한 복제본이 존재하지 않았는데
   "동기화됨"으로 잘못 판단했다.
2. index.md는 `leetaewon-brand-index.md`라는 이름의 복제본이 프로젝트
   지식에 전혀 없었는데도, 검색에 잡힌 저장소 원본 `index.md`를 복제본으로
   착각해 "동기화됨"으로 잘못 판단했다.

두 오판 모두 원인이 같다 — "검색 결과 중 하나를 골라 버전을 읽는다"는
절차가 "저장소 버전과 정확히 일치하는 게 있는지 확인한다"는 절차와
다르다는 점이 스킬에 명시되어 있지 않았다.

추가로, 프로젝트 지식에 GitHub 저장소가 2개 이상 연결된 경우
(`leetaewon-brand`, `claude-context-design-book` 등) 검색 결과에 다른
저장소의 CLAUDE.md/index.md가 함께 섞여 나올 수 있다는 점도 명시되어
있지 않아, 다중 저장소 환경에서 같은 오판이 재발할 위험이 있었다.

### 변경 내용

**1-1. 저장소 소속 필터링 추가**
검색 결과 각 청크의 본문에 명시된 저장소명이 지금 다루는 `[저장소명]`과
정확히 일치하는지 확인하고, 일치하지 않는 청크(다른 저장소 것)는 완전히
무시하도록 했다. 일치하는 청크가 없으면 "찾지 못함"으로 처리하고 쿼리를
구체화해 재검색하도록 했다.

**1-2. "복제본 버전 읽기" → "동일 버전 존재 여부 확인"으로 절차 자체를 변경**
목적을 "복제본의 버전이 무엇인지 알아내기"에서 "저장소 버전과 정확히
일치하는 복제본이 있는지 확인하기"로 바꿨다. 검색 결과에 여러 버전이
섞여 나오는 것을 정상으로 간주하고, 모든 버전 태그를 추출한 뒤 저장소
버전과 일치하는 것이 있는지로만 판단하도록 했다. 1-1과 동일한 저장소
소속 필터링도 적용했다.

**1-3. 상태 분류 세분화**
"교체 필요"(복제본은 있으나 동일 버전이 없어 구버전만 존재)와 "복제본만
없음"(복제본이 전혀 없음)을 구분했다. 처리 방식(Step 2)은 동일하지만,
전자만 구버전 삭제 안내가 필요하다는 차이를 반영했다.

**Step 4. 안내 문구 보강**
구버전이 함께 발견된 경우, "동일 버전이 없어서 새로 만들었으니 예전
버전이 남아있다면 삭제해 달라"는 안내가 출력되도록 문구를 변경했다.

### 영향

- 다중 저장소가 연결된 프로젝트에서 다른 저장소의 CLAUDE.md/index.md를
  잘못 비교 대상으로 삼는 오류를 방지한다.
- 복제본이 여러 버전 뒤섞여 존재하는 상태에서 "동기화됨"으로 잘못
  판단하는 오류를 방지한다.
- 사용자가 구버전 정리 시점을 명확히 안내받을 수 있다.
