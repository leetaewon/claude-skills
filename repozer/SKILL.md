# Repozer — 정보형 저장소 문서 관리기

`/repozer [저장소명]` 명령어로만 실행된다. 다른 표현에는 반응하지 않는다.

---

## 관리 대상 파일

repozer는 저장소의 두 파일을 관리한다. 두 파일은 **독립적으로 버전 관리**된다.

| 파일 | 역할 | 업데이트 시점 |
|---|---|---|
| `CLAUDE.md` | 저장소 목적, Claude 접근 방식 | 목적·방식이 바뀔 때 (드묾) |
| `index.md` | 폴더/파일 구조 + 한 줄 설명 | 파일·폴더 추가·변경될 때 |

**버전 형식**: `<!-- version: YYYY-MM-DD-NNN -->`

**프로젝트 지식 복제본 네이밍**:
- `CLAUDE.md` → `{저장소명}-CLAUDE.md`
- `index.md` → `{저장소명}-index.md`

---

## ⚠️ 파일 접근 방식 (반드시 준수)

저장소 파일은 **Claude 프로젝트 지식**에 직접 연결되어 있다.

- `web_fetch`, GitHub API, 웹 검색으로 저장소를 찾으려 하지 않는다
- 반드시 `project_knowledge_search` 도구로 저장소 내용을 검색한다
- 프로젝트 지식에서 찾을 수 없으면 → 외부 접근을 시도하지 말고 사용자에게 안내한다:
  ```
  [저장소명] 관련 파일을 프로젝트 지식에서 찾을 수 없습니다.
  해당 GitHub 저장소가 이 프로젝트의 지식에 추가되어 있는지 확인해주세요.
  ```

---

## 대상 저장소

repozer는 **정보형 저장소**만 다룬다.

> 정보형 저장소 = Claude가 저장소를 잘 찾고 참조하게 하는 것이 목적인 저장소.
> 트리거·프로세스·행동 원칙이 필요 없는 저장소.

**정보형 신호**
- 폴더 구조가 주제·카테고리별로 구분됨
- 프로세스·단계·절차를 나타내는 파일명이 없음
- README가 "무엇이 있는가"를 설명함

저장소를 읽었을 때 프로세스형 신호(단계별 폴더, 트리거 표현, 순서가 있는 파일명 등)가 감지되면:
```
⚠️ [저장소명] — 프로세스 포함 저장소로 파악됩니다.

감지된 신호: [감지된 신호 한 줄 설명]

repozer는 정보형 저장소를 기준으로 CLAUDE.md와 index.md를 생성합니다.
트리거·프로세스 맵·행동 원칙 섹션은 포함되지 않습니다.

그래도 계속 진행할까요?
```

사용자가 동의하면 → Step 1(진단)으로 진행.
사용자가 거절하면 → 종료.

---

## Step 1. 진단 — 버전 비교

`/repozer [저장소명]` 실행 시 아래 순서로 `project_knowledge_search`를 호출한다.

### 1-1. 저장소의 두 파일 읽기

두 파일만 읽는다. 저장소 전체 탐색은 하지 않는다.

```
project_knowledge_search("[저장소명] CLAUDE.md")
project_knowledge_search("[저장소명] index.md")
```

⚠️ **저장소 소속 필터링 (필수)**

프로젝트 지식에 GitHub 저장소가 2개 이상 연결된 경우, 검색 쿼리에
`[저장소명]`을 넣어도 **다른 저장소의 CLAUDE.md/index.md가 같은 검색
결과에 함께 섞여 나올 수 있다.** 모든 CLAUDE.md는 제목이 똑같이
`CLAUDE.md`로 표시되고 첫 줄 형식도 `# CLAUDE.md — {저장소명}`으로
동일하므로, 제목만 보고 "이건 [저장소명]의 파일이다"라고 단정하지 않는다.

각 검색 결과 청크에 대해 반드시 다음을 확인한다:
- 본문 1~3줄 안에 명시된 저장소명(`# CLAUDE.md — {저장소명}` 또는
  "이 저장소가 무엇인가" 섹션의 저장소명, `index.md`의 `# index —
  {저장소명}`)이 지금 다루는 `[저장소명]`과 **정확히 일치하는지** 확인한다.
- 일치하지 않는 청크(다른 저장소의 CLAUDE.md/index.md)는 **완전히 무시**한다.
  버전 비교에도, 신규 생성 판단에도 사용하지 않는다.
- 같은 검색 결과 안에 `[저장소명]`이 명시된 청크가 하나도 없으면, 그
  검색에서는 "찾지 못함"으로 처리하고 쿼리를 더 구체화해서 재검색한다
  (예: `"[저장소명] 저장소 CLAUDE.md"`처럼 저장소명을 본문에 등장하는
  형태와 더 가깝게 만든다).

두 검색에서 각각, **[저장소명]이 확인된 청크만**을 대상으로
`<!-- version: ... -->` 태그를 추출한다.

- 파일이 존재하고 버전 태그가 있으면 → 버전값 기록
- 파일이 존재하지 않거나 버전 태그가 없으면 → "없음"으로 기록

### 1-2. 프로젝트 지식 복제본 — "동일 버전 존재 여부" 확인

⚠️ 이 단계의 목적은 "복제본의 버전이 무엇인지 알아내는 것"이 **아니다**.
목적은 **"1-1에서 확인한 저장소 버전과 정확히 일치하는 복제본이 프로젝트
지식에 이미 존재하는가"**를 확인하는 것이다. 이 둘은 다르다 — 전자는 검색
결과 중 하나를 임의로 골라야 하지만, 후자는 일치 여부만 boolean으로
판단하면 된다.

```
project_knowledge_search("[저장소명]-CLAUDE.md")
project_knowledge_search("[저장소명]-index.md")
```

**검색 결과 처리 원칙**
- 1-1과 동일하게, **다른 저장소의 복제본이 섞여 나올 수 있다**
  (`leetaewon-contents-CLAUDE.md`, `claude-context-design-book-CLAUDE.md`
  등). 파일명이 `[저장소명]-CLAUDE.md`와 다르게 표시되는 청크는 즉시
  제외한다. 파일명이 같아 보여도 본문 저장소명이 다르면 제외한다.
- 검색 결과에는 **여러 버전의 청크가 동시에 섞여 나올 수 있다** (옛날 버전,
  버전 태그 없는 버전, 다른 파일명으로 저장된 옛 버전 등). 이는 정상이며
  오류가 아니다.
- 결과에 등장하는 **모든 버전 태그를 빠짐없이 추출**한다. 가장 그럴듯해
  보이는 것 하나만 골라서 "이게 복제본 버전이다"라고 단정하지 않는다.
- 추출한 버전 목록 중에 **1-1의 저장소 버전과 정확히 일치하는 것이 있는지**
  확인한다.
  - 일치하는 것이 있으면 → "동일 버전 존재" (그 외 버전들은 구버전 잔존물로
    별도 기록)
  - 일치하는 것이 없으면(전부 다른 버전이거나, 검색 결과 자체가 없으면) →
    "동일 버전 없음"
- 제목 표시(`[저장소명]-CLAUDE.md` vs `CLAUDE.md`)만으로 저장소 원본과
  복제본을 구분하지 않는다. 본문의 "저장소 맵" 섹션이 `index.md`를
  가리키는지 `[저장소명]-index.md`를 가리키는지로 원본/복제본을 구분한다.

### 1-3. 상태 판단

아래 상태를 각 파일에 대해 독립적으로 판단한다.

| 상태 | 조건 |
|---|---|
| **동기화됨** | 저장소 파일 존재 + 동일 버전의 복제본 존재 |
| **교체 필요** | 저장소 파일 존재 + 복제본은 존재하나 동일 버전이 없음 (구버전만 존재) |
| **복제본만 없음** | 저장소 파일 존재 + 복제본이 전혀 검색되지 않음 |
| **신규 생성 필요** | 저장소 파일 자체가 없음 |

"교체 필요"와 "복제본만 없음"은 처리 방식(Step 2)이 동일하지만(저장소
내용으로 신규 출력), 안내 문구에서는 구분해서 출력한다 — 전자는 구버전
삭제 안내가 필요하고 후자는 필요 없다.

판단 결과를 출력한다:

```
📋 [저장소명] 진단 결과

CLAUDE.md:  저장소 버전 2026-06-16-001 / 복제본 버전 2026-05-10-001 → 교체 필요
index.md:   저장소 버전 없음 / 복제본 없음 → 신규 생성 필요
```

→ 바로 Step 2로 진행한다.

## Step 2. 처리

판단 결과에 따라 각 파일을 독립적으로 처리한다.

### 케이스별 처리

| 상태 | 처리 |
|---|---|
| 동기화됨 | 건너뜀 (아무것도 하지 않음) |
| 교체 필요 | 저장소의 파일 내용으로 복제본 교체 출력 |
| 복제본만 없음 | 저장소의 파일 내용 그대로 복제본 출력 |
| 신규 생성 필요 | → **Step 3(신규 생성)**으로 분기 |

### 교체·복제본 출력

저장소 파일의 내용을 아티팩트로 출력한다.
- 아티팩트 제목: `[저장소명]-CLAUDE.md` 또는 `[저장소명]-index.md`
- 내용은 저장소 파일과 동일하게 출력하되, **저장소 맵 섹션 한 줄만 치환**한다

**경로 치환 규칙 (CLAUDE.md에만 적용, 한 줄 한정)**

`CLAUDE.md`의 "저장소 맵" 섹션은 동료 파일 `index.md`를 가리키는 유일한
네비게이션 지시다. 저장소 안에서는 `index.md`가 맞는 이름이지만, 프로젝트
지식 복제본 안에서는 그 파일이 `{저장소명}-index.md`라는 이름으로 존재하므로
이 섹션만 치환한다.

```
## 저장소 맵

→ `index.md` 참조
```

```
## 저장소 맵

→ `[저장소명]-index.md` 참조
```

**다른 곳은 손대지 않는다.** `CLAUDE.md` 본문 중 "운영 원칙", "파일 업데이트
시" 등에 나오는 `index.md`라는 표현은 "GitHub 저장소 안의 index.md 파일을
갱신하라"는 작업 지침이며 지금 이 복제본 옆 파일을 가리키는 네비게이션이
아니므로 그대로 둔다. `index.md` 파일 자체에는 치환 대상이 없다 — 트리 안의
`CLAUDE.md`/`index.md` 표기와 하단 서명은 모두 저장소 안에서의 실제 파일명을
설명하는 것이므로 원본 그대로 출력한다.

저장소 전체를 스캔해서 `index.md`나 `CLAUDE.md`라는 글자가 나오는 곳을 찾아
바꾸지 않는다. 치환은 "저장소 맵" 섹션 한 줄로 한정한다.

## Step 3. 신규 생성 (파일이 없는 경우)

**CLAUDE.md 또는 index.md가 저장소에 없을 때만** 이 단계를 실행한다.

### 3-1. 저장소 탐색

저장소 내용을 파악하기 위해 `project_knowledge_search`를 반복 호출한다.

**검색 순서**
1. `[저장소명] README` → 저장소 개요, 목적, 전체 폴더 구조 파악
2. `[저장소명] 파일 구조` → 추가 파일 확인
3. 1~2에서 발견된 폴더명·파일명을 키워드로 추가 검색 → 각 파일 내용 파악

3회 이상 검색해도 파악이 안 되면 → 사용자에게 안내 후 종료:
```
[저장소명] 저장소 내용을 프로젝트 지식에서 충분히 파악할 수 없습니다.
저장소 파일이 이 프로젝트 지식에 추가되어 있는지 확인해주세요.
```

### 3-2. CLAUDE.md 생성

아티팩트 제목: `[저장소명]-CLAUDE.md`

**작성 규칙**
- 아래 섹션 목록을 그대로 따른다. 없는 섹션을 임의로 추가하지 않는다
- 고정 텍스트 섹션은 명시된 내용을 그대로 복사한다
- 파악된 내용으로만 채운다. 파악 못한 내용은 placeholder 처리:
  ```
  <!-- TODO: [무엇이 필요한지] -->
  ```

**섹션 목록**

| 섹션 | 유형 |
|---|---|
| 헤더 (`# CLAUDE.md — {저장소명}` + 버전 주석) | 파악된 내용으로 채움 |
| 저장소 목적 | 파악된 내용으로 채움 |
| Claude 접근 방식 | **고정 텍스트** |
| 연결 저장소 | 파악된 내용으로 채움 (없으면 섹션 생략) |

**고정 텍스트: Claude 접근 방식**

```markdown
## Claude 접근 방식

이 저장소의 파일들은 Claude 프로젝트 지식에 직접 연결되어 있다.
파일 내용 참조 시 web_fetch나 GitHub MCP를 사용하지 말고,
프로젝트 지식에서 직접 참조한다.

저장소 맵은 [저장소명]-index.md를 참조한다.
```

> 위 고정 텍스트의 마지막 줄("저장소 맵은 ... 참조한다")이 바로 그 한 줄이다.
> 복제본(`[저장소명]-CLAUDE.md`)에는 위처럼 `[저장소명]-index.md`로 적고,
> 저장소 루트의 원본 `CLAUDE.md`에 커밋할 때는 `index.md`로 되돌린다 —
> 저장소 안에서는 실제 파일명이 `index.md`이기 때문이다. 고정 텍스트의
> 다른 줄에는 치환 대상이 없다.

**버전 형식**

헤더 바로 아래에 작성:
```
# CLAUDE.md — {저장소명}
<!-- version: YYYY-MM-DD-001 -->
```

### 3-3. index.md 생성

아티팩트 제목: `[저장소명]-index.md`

**작성 규칙**
- 탐색으로 파악한 실제 구조만 반영한다
- 파악 못한 파일/폴더는 `<!-- TODO -->` 처리
- 핵심 파일 추정에는 ★ 표시

**섹션 목록**

| 섹션 | 유형 |
|---|---|
| 헤더 (`# index — {저장소명}` + 버전 주석) | 파악된 내용으로 채움 |
| 저장소 구조 (트리) | 파악된 내용으로 채움 |
| 파일/폴더 설명 표 | 파악된 내용으로 채움 |

**표 형식**

```markdown
| 경로 | 내용 | 참조 시점 |
|---|---|---|
| `폴더명/` | 설명 | 언제 참조하나 |
| `파일명.md` | 설명 | 언제 참조하나 |
```

**버전 형식**

```
# index — {저장소명}
<!-- version: YYYY-MM-DD-001 -->
```

## Step 4. 결과 안내

### 모두 동기화됨

```
✅ [저장소명] — 모두 최신 상태

CLAUDE.md: 동기화됨 (2026-06-16-001)
index.md:  동기화됨 (2026-06-16-001)
```

### 교체/복제본 출력이 있는 경우 — 구버전이 함께 발견된 경우

```
🔄 [저장소명] — 업데이트 완료

CLAUDE.md: 동일 버전(2026-06-16-002) 복제본 없음 → 신규 생성
           ⚠️ 구버전 [N]개 발견 (버전 태그 없음 / 2026-06-09 등)
index.md:  동기화됨 (건너뜀)

다음 단계:
① 위 아티팩트를 [저장소명]-CLAUDE.md로 저장 후 프로젝트 지식에 추가
② 프로젝트 지식에서 위 구버전들을 찾아 삭제 — 동일 버전이 없어서 새로
   만들었으니, 예전 버전이 남아있다면 삭제해 주세요
```

### 신규 생성이 있는 경우 (저장소 자체에 파일 없음)

```
🆕 [저장소명] — 신규 파일 생성

CLAUDE.md: 신규 생성 <!-- TODO --> [N]곳
index.md:  신규 생성 <!-- TODO --> [N]곳

다음 단계:
① 위 아티팩트를 각각 저장소 루트에 업로드
   - [저장소명]-CLAUDE.md → 저장소의 CLAUDE.md
   - [저장소명]-index.md  → 저장소의 index.md
② 프로젝트 지식에 추가 ('연결'이 아닌 '추가')
③ <!-- TODO --> 항목은 내용을 알려주시면 채워드립니다
```

### 혼합 (교체 + 신규 생성)

위 두 형식을 합쳐서 출력한다.
